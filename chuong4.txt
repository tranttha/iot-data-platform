CHƯƠNG 4: TRIỂN KHAI CHƯƠNG TRÌNH

═══════════════════════════════════════════════════════════════
4.1. CẤU TRÚC THƯ MỤC VÀ TỔNG QUAN TRIỂN KHAI
═══════════════════════════════════════════════════════════════

Toàn bộ dự án được tổ chức theo mô hình monorepo, mỗi thư mục con tương ứng với một thành phần
độc lập trong kiến trúc pipeline:

iot-data-platform/
├── docker-compose.yaml          # Điều phối toàn bộ stack, chia profile theo tầng
├── .env                         # Biến môi trường (credentials, resource limits)
├── iot-gateway/                 # Producer giả lập dữ liệu IoT
├── kafka-spark-consumer/        # Spark job: Kafka → MinIO /raw
├── spark-stream/                # Spark job: /raw → Iceberg warehouse
├── trino/                       # Cấu hình Trino query engine
├── grafana/                     # Cấu hình Grafana provisioning
└── minio-data/                  # Bind-mount volume lưu trữ MinIO

Stack được chia thành 4 profile docker-compose:
  - ingest:     Kafka + IoT Gateway + kafka-spark-consumer → ghi dữ liệu thô vào MinIO
  - processing: spark-stream → đọc /raw, xử lý, ghi vào Iceberg warehouse
  - serve:      Trino + Grafana → truy vấn và trực quan hóa
  - full:       Toàn bộ pipeline end-to-end

Spark standalone cluster (1 master + 1 worker) được chia sẻ giữa hai profile ingest và
processing. Worker được cấp 9 cores và 5 GB RAM: kafka-spark-consumer sử dụng 3 cores,
spark-stream sử dụng 6 cores, đảm bảo cả hai job chạy song song không tranh chấp tài nguyên.


───────────────────────────────────────────────────────────────
4.2. IOT GATEWAY — PRODUCER GIẢ LẬP (iot-gateway/)
───────────────────────────────────────────────────────────────

Thành phần này đóng vai trò nguồn dữ liệu cho toàn bộ pipeline. Thay vì gọi API thực của
OpenWeatherMap (bị giới hạn 1.000 calls/ngày ở free tier, tối đa cập nhật 10 phút/lần), hệ
thống triển khai một bộ sinh dữ liệu giả lập (data simulator) có schema hoàn toàn tương đồng
với OpenWeatherMap One Call 3.0 API, nhưng tăng tần suất sinh bản ghi tùy ý để kiểm thử
thông lượng cao.

── Schema payload ────────────────────────────────────────────

  Mỗi message gửi vào Kafka có cấu trúc:
  {
    "iot_device_id":  1,
    "ingestion_time": "2026-05-07T10:00:01Z",
    "data": {
      "dt": 1746612001,           // Unix timestamp (event time)
      "temp": 22.5,               // Nhiệt độ (°C)
      "feels_like": 21.1,
      "pressure": 1013,           // Áp suất (hPa)
      "humidity": 65,             // Độ ẩm (%)
      "dew_point": 15.3,
      "uvi": 3.2,
      "clouds": 40,
      "visibility": 8000,
      "wind_speed": 4.5,          // Tốc độ gió (m/s)
      "wind_deg": 210,
      "wind_gust": 7.1,
      "weather": [{"id": 801, "main": "Clouds", "description": "few clouds"}],
      "rain": {"1h": 0.5}         // Optional
    },
    "location": {
      "lat": 40.71, "lon": -74.01, "timezone": "America/New_York"
    }
  }

── Cơ chế hoạt động ──────────────────────────────────────────

  Một container duy nhất quản lý 60 trạm khí tượng được định nghĩa trong config.yaml, trải
  rộng trên 6 châu lục. Producer chạy vòng lặp liên tục, gửi lần lượt 1 trạm/giây — tức 60
  bản ghi mỗi phút. Dữ liệu nhiệt độ được sinh dựa trên mô hình khí hậu theo vĩ độ (nhiệt độ
  giảm ~0.55°C mỗi độ vĩ độ) cộng thêm nhiễu Gaussian, đảm bảo tính thực tế của chuỗi thời
  gian.

  Message key được đặt bằng iot_device_id để Kafka hash cố định mỗi thiết bị vào một
  partition duy nhất, bảo toàn thứ tự sự kiện theo từng thiết bị.


───────────────────────────────────────────────────────────────
4.3. KAFKA-SPARK-CONSUMER — INGEST VÀO MINIO (kafka-spark-consumer/)
───────────────────────────────────────────────────────────────

Job Spark Structured Streaming này liên tục đọc message từ Kafka topic iot-sensor-data và
ghi nguyên vẹn (không transform) xuống vùng raw trong MinIO dưới định dạng JSON, phân vùng
theo ngày và thiết bị:

  s3a://raw/ingestion_date=2026-05-07/iot_device_id=1/<part-*.json>

Cấu hình spark-submit sử dụng 3 executor cores (= số partition Kafka), đảm bảo mỗi partition
Kafka được xử lý bởi một Spark task riêng biệt, tối ưu hóa throughput.

Checkpoint được lưu tại s3a://raw/_checkpoints/ trên MinIO, đảm bảo exactly-once semantics:
khi container restart, job tiếp tục đọc đúng từ offset đã commit, không mất hay trùng dữ liệu.

JAR dependency (Kafka connector, hadoop-aws, aws-java-sdk) được cache trong Docker volume
ivy2_cache để tránh tải lại từ Maven Central mỗi lần restart.


───────────────────────────────────────────────────────────────
4.4. SPARK-STREAM — XỬ LÝ VÀ MÔ HÌNH HÓA DỮ LIỆU (spark-stream/)
───────────────────────────────────────────────────────────────

Đây là thành phần trung tâm của pipeline, được thiết kế theo mô hình ephemeral container:
thay vì chạy như một long-running streaming job, spark-stream khởi động → xử lý toàn bộ
dữ liệu mới → dừng hẳn → ngủ POLL_INTERVAL_SECONDS giây → lặp lại từ đầu. Mỗi lần chạy
là một "lô mini" (mini-batch) độc lập, xử lý đúng phần dữ liệu đã xuất hiện trong /raw kể
từ checkpoint trước.

Cơ chế này được thực hiện thông qua vòng lặp shell trong CMD của Dockerfile:

  while true; do
      spark-submit ... /app/stream.py   # chạy job, blocks đến khi hoàn tất
      sleep ${POLL_INTERVAL_SECONDS}    # ngủ 60 giây
  done

Bên trong mỗi lần chạy, Spark dùng trigger(availableNow=True): query tự dừng sau khi đã xử
lý hết file hiện có, không chờ dữ liệu mới. Kết hợp với checkpoint, mỗi mini-batch chỉ đọc
đúng các file được ghi bởi kafka-spark-consumer trong 60 giây vừa qua.

So sánh với mô hình long-running streaming:
  - Long-running: 1 JVM tồn tại mãi, nhận micro-batch liên tục, tiêu tốn tài nguyên kể cả
    lúc không có dữ liệu, khó restart sạch.
  - Ephemeral (mô hình này): JVM chỉ tồn tại trong thời gian xử lý (~30–60 giây/chu kỳ),
    sau đó giải phóng hoàn toàn tài nguyên Spark worker. Tương đương với Kubernetes CronJob
    hoặc scheduled task, dễ giám sát trạng thái (exit 0 = thành công, exit ≠ 0 = lỗi).

Mỗi chu kỳ 60 giây thực hiện đầy đủ 10 bước theo thứ tự:

  1. Đọc tất cả file JSON mới từ s3a://raw/ (Spark Structured Streaming).
  2. Flatten schema OWM: trích xuất các trường từ cấu trúc JSON lồng nhau thành bản ghi phẳng.
  3. Dedup trong batch: dropDuplicates(event_time, iot_device_id).
  4. MERGE INTO staging.events (Iceberg): dedup xuyên run (cross-batch deduplication).
  5. Upsert dim_date: thêm các ngày mới từ batch vào bảng chiều thời gian.
  6. SCD2 merge dim_iot_device: cập nhật lịch sử thay đổi thông tin thiết bị.
  7. SCD2 merge dim_country: cập nhật lịch sử thay đổi thông tin quốc gia.
  8. INSERT INTO fact_measurements: nạp bản ghi đo lường với khóa ngoại tới các chiều.
  9. MERGE INTO fact_hourly_agg: tái tính tổng hợp theo giờ cho 2 giờ gần nhất.
  10. Compact small files: gọi Iceberg rewrite_data_files() cho 2 ngày gần nhất.

── Mô hình dữ liệu Iceberg (warehouse/) ─────────────────────

  Tất cả bảng lưu trữ dưới dạng Apache Iceberg trên MinIO (s3a://warehouse/), sử dụng
  Hadoop catalog (file-based, không cần metastore). Trino đọc trực tiếp từ cùng warehouse
  path.

  Bảng              Loại      Phân vùng            Mô tả
  ────────────────  ────────  ───────────────────  ──────────────────────────────────────────
  staging.events    staging   days(event_time)     Bản ghi thô đã flatten, dùng để dedup
  dim_date          dimension —                    Chiều thời gian (ngày, tháng, quý, năm)
  dim_iot_device    dimension —                    Chiều thiết bị IoT (SCD Type 2)
  dim_country       dimension —                    Chiều quốc gia/khu vực (SCD Type 2)
  fact_measurements fact      days(event_time)     Bảng sự kiện chính với FK tới các chiều
  fact_hourly_agg   aggregate hours(window_start)  Tổng hợp theo giờ (avg temp, avg humidity…)

── Tài nguyên Spark ──────────────────────────────────────────

  spark-stream sử dụng 6 executor cores và 512 MB executor memory. Toàn bộ 6 cores được
  gom vào một executor duy nhất (--conf spark.executor.cores=6) để giữ memory footprint trong
  ngưỡng 512 MB thay vì nhân lên 6 × 512 MB.

── Điểm đáng chú ý ───────────────────────────────────────────

  - minio-client (mc): init container chạy một lần, tự động tạo 3 bucket (raw, staging,
    warehouse) rồi thoát. Pattern tương tự InitContainers trong Kubernetes.

  - ivy2_cache: named volume mount vào /root/.ivy2 của kafka-spark-consumer và spark-stream,
    cache JAR Maven (~200 MB) để tránh tải lại khi restart container.

  - MinIO dùng bind-mount (./minio-data:/data) thay vì named volume để dữ liệu tồn tại
    trực tiếp trên filesystem host, dễ kiểm tra và sao lưu.

  - spark.driver.host được set tường minh trong CMD của mỗi Dockerfile (kafka-spark-consumer,
    spark-stream) để worker trong Docker network biết địa chỉ driver để trả kết quả task về.


───────────────────────────────────────────────────────────────
4.6. TRINO VÀ GRAFANA — TẦNG PHỤC VỤ (serve)
───────────────────────────────────────────────────────────────

Trino được cấu hình với Iceberg connector trỏ trực tiếp vào warehouse path trên MinIO
(s3a://warehouse/). File cấu hình catalog (trino/catalog/iceberg.properties) khai báo
hive.metastore=file, cho phép Trino tự đọc metadata từ filesystem mà không cần Hive Metastore.
Người dùng có thể thực hiện SQL ad-hoc trực tiếp trên dữ liệu Iceberg qua cổng 8080.

Grafana kết nối vào Trino qua plugin Trino data source (JDBC), cho phép xây dựng dashboard
near real-time với dữ liệu mới nhất từ fact_measurements và fact_hourly_agg. Cấu hình
provisioning dashboard và datasource được lưu trong grafana/provisioning/ để tự động load
khi container khởi động.


───────────────────────────────────────────────────────────────
4.7. THÔNG LƯỢNG VÀ ĐỘ TRỄ CỦA PIPELINE
───────────────────────────────────────────────────────────────

── Thông lượng sinh dữ liệu (Producer) ──────────────────────

  Producer vòng lặp liên tục qua 60 trạm, sleep 1 giây giữa mỗi trạm:

    Tần suất sinh bản ghi  = 60 trạm ÷ 60 giây = 1 msg/giây = 60 msg/phút
    Kích thước mỗi message ≈ 500 – 700 bytes (JSON với ~22 trường số)
    Băng thông Kafka ingest ≈ 1 msg/s × 600 B ≈ 600 B/s ≈ 2.1 MB/giờ

  Lượng dữ liệu tích lũy theo thời gian:

    Khoảng thời gian   Số bản ghi     Dung lượng raw (ước tính)
    ───────────────    ────────────   ─────────────────────────
    1 phút             60 msg         ~36 KB
    1 giờ              3.600 msg      ~2.1 MB
    1 ngày             86.400 msg     ~50 MB
    1 tuần             604.800 msg    ~350 MB

── Thông lượng xử lý theo tầng ──────────────────────────────

  Tầng 1 — kafka-spark-consumer (Kafka → MinIO /raw):
    Trigger mỗi POLL_INTERVAL_SECONDS = 600 giây.
    Mỗi micro-batch xử lý ≈ 600 messages (60 msg/phút × 10 phút).
    Thời gian xử lý mỗi batch: < 10 giây (3 Spark tasks song song, mỗi task
    xử lý 1 Kafka partition với ~200 msg).
    Utilization: ~10 giây xử lý / 600 giây chu kỳ ≈ 1.7% → rất nhiều headroom.

  Tầng 2 — spark-stream (/raw → Iceberg warehouse):
    Trigger mỗi POLL_INTERVAL_SECONDS = 600 giây với trigger=availableNow.
    Mỗi run xử lý toàn bộ file JSON mới trong /raw kể từ checkpoint trước (~600 rows).
    Thực hiện 10 bước pipeline (flatten → dedup → 4 bảng warehouse → agg → compact).
    Thời gian xử lý ước tính: 30 – 90 giây / run tùy kích thước batch và cold-start JVM.
    Utilization: ~60 giây xử lý / 600 giây chu kỳ ≈ 10%.

── Độ trễ end-to-end (Event Time → Visible in Grafana) ──────

  Sự kiện từ khi producer sinh ra tới khi xuất hiện trên dashboard Grafana trải qua
  các bước trễ tích lũy sau:

    Bước                                  Độ trễ ước tính
    ────────────────────────────────────  ──────────────────────
    Producer → Kafka (network + ack)      < 1 giây
    Kafka → MinIO /raw (trigger interval) 0 – 600 giây (trung bình 300 giây)
    MinIO /raw → Iceberg warehouse        0 – 600 giây (trung bình 300 giây)
    Spark processing time                 30 – 90 giây
    Grafana refresh interval              30 giây (mặc định)
    ────────────────────────────────────  ──────────────────────
    Tổng độ trễ (worst case)              ≈ 1.230 giây ≈ 20 phút
    Tổng độ trễ (trung bình)              ≈ 660 giây  ≈ 11 phút

  Đây là đặc tính của kiến trúc micro-batch: độ trễ tỉ lệ thuận với trigger interval.
  Giảm POLL_INTERVAL_SECONDS xuống (VD: 60 giây) có thể kéo độ trễ trung bình xuống còn
  ~2 phút, nhưng sẽ tăng số lượng small files trong MinIO và overhead checkpoint.

── Khả năng mở rộng ──────────────────────────────────────────

  Pipeline hiện tại chạy trên 1 Spark worker với 9 cores. Để tăng thông lượng:

    Tăng số trạm: thêm stations vào config.yaml (không cần rebuild image).
    Tăng Kafka partitions (KAFKA_NUM_PARTITIONS) và CONSUMER_TOTAL_EXECUTOR_CORES
      tương ứng để tăng song song hóa tầng ingest.
    Thêm Spark worker: docker-compose scale spark-worker=2, Spark tự phân bổ task.
    Giảm POLL_INTERVAL_SECONDS: tăng tần suất xử lý, giảm độ trễ end-to-end.
